## 1. Использование

#### part_1

Содержит две папки `src` и  `log` в первой находится основной скрипт `main.sh`, который выполняет запрос у
указанному ресурсу и выполняет логирование по следующему пути `/part_1/log/log.txt`

Использование 

``` bash
chmod +x main.sh
./main.sh 
```
#### part_2

Содержит `Docekrfile` в котором осущетсвляется сборка и запукс контейнера со скриптом `main.sh`,
Также в директории есть скрипт `cheks_log.sh` с выводом лог файала 

Использование
    
``` bash
podman build -f ../part_2/Dockerfile -t my-app .
podman run my-app
```
Для вывода лог файла
```
chmod +x checks_log.sh
./checks_log.sh
```
#### part_3 

Содержит Ansible playbook и inventori файл с инструкциями для работы с предыдущми артефактами проекта
Также находится скрипт `get-docker.sh` по устновке докера на целевую систему

Использование ansible 
```bash
ansible-playbook -i inventory.ini deploy.yml
```

Для использования нужно сгенерировать пару ssh ключей и добавить публичный на целевую машину
```bash
ssh-keygen -t rsa -b 4096 
ssh-copy-id user@vm_ip
```

Затем в целях безопасности в директории `/.ssh` создать директорию `config` и добавить следующие значения
``` yaml
Host vm1 -- здесь вы можете указать любое удобное значение, которое будет в invenori файле1
    HostName 192.168.1.100 -- целевое IP 
    User ubuntu -- имя пользователя с помощью которого будет осуществляться подключениеu
    IdentityFile ~/.ssh/ansible_key
```
Это было сделано, чтобы скрыть закрытый ключ и не использовать его явно в файле инвенторя в целях безопасности

## 2. Процесс решения

####  P.S

Так как на хосте осоновоное средство виртуализации `podman`, то во всех скрпитах испольуется именно
этот истркмент, однако все решения совметимы и с `docker`

#### part_1

При решении первой части я решил разделить логику осущетсвления запросов и логирования
Для этого я создал две разные директории, так как мне показалось это удобным с точки зрения 
Масштабируемости, однако с этим в будущем возникли проблемы, 
Так как логи пишутся в отдельный файл после запуска контейнера, то вывод 
`podman logs` будерт пустым, так как контейнер после исполнения скрипта завершается со статус кодом 0 , то
остается выгрузить логи на хост командой `cp` и просмотреть их с помощью `cat`, что и было реализовано 
во второой части

#### part_2

Главной целью было создать легкий образ, поэтому я решил использовать только необходимые команды 
для сборки образа, чтобы не создавать лишние слои, для этой же цели установка нескольких зависимотей 
были указаны в одной `RUN` команде, а также в целях безопасности использовад  `COPY` вместо `ADD` для переноса файла


#### part_3

Изначально хотел использовать задачу c synchronize, чтобы избежать нескольких этапов копирования, но в итоге не вышло(возможно из-за того что я не првильно задал сируктуру для целевой директории), поэтому в итоге добавил более прямолинейную структуру, файл get-docker.sh был найден мной давно, когда я менял `podman` на `docker` на одном из RedHat дистрибутивов на ВМ, тогда он вполнился умпешно, поэтомк решил заново его использовать и добавить в репозиторий
